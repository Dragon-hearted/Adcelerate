# Domain Knowledge — ReadmeEngine

## Core Domain
Technical documentation generation — specifically README.md files that serve as the primary entry point for developers encountering a project. The domain sits at the intersection of software documentation, developer experience (DX), and information architecture. Key concepts:

- **README as product**: A README is the storefront of a project. It must answer "what is this?", "why should I care?", and "how do I use it?" within the first screen.
- **Knowledge-driven generation**: Rather than static templates, READMEs are generated by reading live knowledge sources (YAML registries, package.json, git history, knowledge/ directories) so they reflect current reality.
- **Drift detection**: Documentation rots faster than code. A fingerprinting system tracks when knowledge sources change and flags stale sections.
- **Scope-aware templates**: Different scopes (monorepo root, individual system, app) need different README structures while maintaining visual consistency.

## Process Knowledge

### Generation Pipeline
1. **Scope parsing** — Determine target(s): root, specific system, specific app, or all
2. **Data collection** — Run collectors to gather structured data from knowledge sources
3. **Section rendering** — Pass collected data through renderers to produce markdown sections
4. **Template assembly** — Arrange sections according to scope-appropriate template
5. **Output writing** — Write assembled README.md to target path

### Collectors (data sources)
- **SystemCollector**: `systems.yaml` + per-system `knowledge/index.md`, `domain.md`, `package.json`
- **GraphCollector**: `knowledge/graph.yaml` + `graphify-out/GRAPH_REPORT.md`
- **LibraryCollector**: `library.yaml` (skills, agents, commands catalog)
- **GitCollector**: `git log` globally and per-target path
- **CodeCollector**: Filesystem scan of `src/` directories for entry points and structure
- **AppCollector**: `apps/*/package.json` and directory structure

### Renderers (markdown producers)
- **Badges**: shields.io badge markdown from package.json dependencies
- **TOC**: Table of contents with emoji-prefixed anchor links
- **Mermaid**: Dependency topology, pipeline stages, platform layer diagrams
- **Architecture**: ASCII box-drawing diagrams from stage definitions
- **TechStack**: Frontend/backend technology tables from package.json
- **ProjectStructure**: Directory tree views with descriptions
- **ApiReference**: API endpoint tables parsed from Hono route files
- **Changelog**: Recent changes section from git log
- **GifReferences**: Demo GIF `<img>` tags from demo/ directories

## Quality Signals

### Visual Quality (from Pinboard README analysis)
- **Centered hero header** with emoji, project name, tagline, and badge row inside `<div align="center">`
- **Horizontal rules** (`---`) between major sections for visual breathing room
- **Emoji-prefixed section headers** (e.g., `## ✨ Features`, `## 🏗 Architecture`)
- **Tables for structured data** — features, tech stack, API endpoints, environment variables
- **Code blocks with language hints** for shell commands, JSON, directory trees
- **Collapsible details** (`<details>`) for verbose content like request/response bodies
- **Consistent spacing** — no cramped sections, no excessive whitespace

### Content Quality
- **Accurate technology versions** — read from actual package.json, not hardcoded
- **Real architecture diagrams** — generated from graph.yaml and systems.yaml data
- **Actionable getting started** — copy-pasteable commands that actually work
- **Complete project structure** — reflects real filesystem, not an idealized view
- **Meaningful descriptions** — demonstrate understanding of each system's purpose and architecture

### Best-README-Template Patterns
Key patterns from the Best-README-Template repository:
- Logo/banner image at the top for visual identity
- One-liner description immediately below the title
- Badges row for quick project status scanning
- "About The Project" section with screenshot/demo
- "Built With" section listing major frameworks/libraries
- "Getting Started" with prerequisites, installation, usage subsections
- "Roadmap" section with checkbox items
- "Contributing" with step-by-step instructions
- "License" section with link to LICENSE file
- "Contact" and "Acknowledgments" footer sections
- Use of `<p align="right">(<a href="#readme-top">back to top</a>)</p>` for navigation

## Edge Cases & Gotchas

- **Missing knowledge sources**: Not all systems have all knowledge files. Collectors must return empty/default data gracefully — missing data means omitted sections, not crashes.
- **Systems with different structures**: Some systems are CLI tools (autoCaption), some are web apps (Pinboard), some are libraries (ImageEngine). Templates must adapt to what's actually present.
- **Mermaid character escaping**: Node labels containing parentheses, quotes, or special characters break mermaid syntax. All labels must be escaped or wrapped in quotes.
- **Badge version detection**: Not all dependencies in package.json should become badges. Only major framework/runtime dependencies deserve badges (TypeScript, Bun, React, etc.).
- **Git history availability**: Some systems are submodules with separate git histories. Git collector must handle both monorepo-native and submodule paths.
- **Demo directory absence**: Most systems don't have a `demo/` directory yet. GIF renderer must insert placeholder comments, not broken image tags.
- **Large file trees**: Deep project structures should be truncated at a reasonable depth (3-4 levels) to avoid overwhelming the README.

## Tacit Expertise

- The Pinboard README at 377 lines is the gold standard in this monorepo. Every generated README should match its visual weight and information density.
- Badges should follow a consistent color scheme — TypeScript blue (#3178C6), Bun cream (#f9f1e1), React cyan (#61DAFB), MIT license coral (#e07a5f).
- ASCII architecture diagrams (box-drawing characters: `┌ ┐ └ ┘ │ ─ ├ ┤ ┬ ┴ ┼ ▶ ▷`) are preferred over mermaid for single-system architecture because they render in all contexts (GitHub, terminal, plain text).
- Mermaid diagrams are preferred for cross-system dependency topology because they benefit from automatic layout.
- The monorepo root README should be the "sales pitch" — it sells the platform to someone who just landed on the repo. System READMEs are the "user manual" — they help someone who's already decided to work with that system.
- When a system is a submodule (ImageEngine), the README should note that it's independently versioned and link to its own repository.

## Best-README-Template Reference Patterns

Detailed structural analysis of [othneildrew/Best-README-Template](https://github.com/othneildrew/Best-README-Template) (14k+ GitHub stars). These patterns inform how ReadmeEngine assembles sections.

### Document Structure (Top to Bottom)

1. **Anchor tag** — `<a id="readme-top"></a>` at document start for "back to top" navigation
2. **Project shields row** — Reference-style badge links (`[![Badge][shield]][url]`) for contributors, forks, stars, issues, license, social
3. **Project logo** — Centered `<div align="center">` with `<img>` logo (80x80), `<h3>` project title, `<p>` one-line description, action links (Explore docs, View Demo, Report Bug, Request Feature) separated by `&middot;`
4. **Table of Contents** — Collapsible `<details><summary>` with nested `<ol>` containing all section links, sub-items for nested sections (Built With under About, Prerequisites/Installation under Getting Started)
5. **About The Project** — Screenshot/product image, prose description of motivation and value proposition
6. **Built With** — List of major frameworks as shield.io badge images (not plain text), each as `[![Name][badge]][url]`
7. **Getting Started** — Intro paragraph, then subsections:
   - **Prerequisites** — Software requirements with install commands in code blocks
   - **Installation** — Numbered steps with code blocks for each command
8. **Usage** — Code examples, screenshots, links to documentation
9. **Roadmap** — Checkbox list (`- [x]` / `- [ ]`) with nested items for sub-features
10. **Contributing** — Prose about open source value, numbered steps (Fork → Branch → Commit → Push → PR), optional contributor images via contrib.rocks
11. **License** — One line: "Distributed under the X License. See `LICENSE.txt` for more information."
12. **Contact** — Name, social links, email, project link
13. **Acknowledgments** — Bulleted list of helpful resources with links
14. **Reference links footer** — All shield/badge URLs and image references defined at document bottom for clean source

### Key Formatting Patterns

| Pattern | Implementation | ReadmeEngine Adoption |
|---------|---------------|----------------------|
| Back-to-top links | `<p align="right">(<a href="#readme-top">back to top</a>)</p>` after each section | Use for long READMEs (>200 lines), skip for short ones |
| Reference-style badges | `[shield]: https://img.shields.io/...` at bottom | Use inline badges instead (simpler for generated output) |
| Centered hero | `<div align="center">` with `<h3>` and `<p>` | Adopt — matches Pinboard pattern |
| Collapsible TOC | `<details><summary>` wrapping `<ol>` | Use flat emoji-linked TOC instead (Pinboard style, better scannability) |
| Badge images for tech | `[![React][React.js]][React-url]` with shield.io `for-the-badge` style | Use smaller inline badges (Pinboard style: `?logo=react&logoColor=white`) |
| Numbered install steps | Ordered list with code blocks per step | Adopt for Getting Started sections |
| Checkbox roadmap | `- [x]` completed, `- [ ]` planned | Omit — not relevant for auto-generated READMEs |
| HTML comments as section dividers | `<!-- SECTION NAME -->` | Use `---` horizontal rules instead (visible, matches Pinboard) |

### Synthesis: ReadmeEngine Template Hierarchy

Combining Best-README-Template structure with Pinboard visual quality, the optimal section order for generated READMEs:

**Root README**: Hero (centered, badges) → Demo/Screenshot → TOC → Platform Overview → Systems Directory (table) → Architecture (mermaid from graph.yaml) → Tech Stack → Skills/Agents Catalog → Getting Started → Project Structure → Contributing → License → Footer

**System README**: Hero (centered, emoji, badges) → Demo GIFs (if demo/ exists) → TOC → Features (table) → Architecture (ASCII box-drawing) → Tech Stack (tables) → Getting Started (Prerequisites, Install, Configure, Run) → Development (justfile) → API Reference (if Hono) → Project Structure (tree) → Configuration (env vars table) → Contributing → License → Footer

**App README**: Hero (centered, badges) → TOC → Overview → Tech Stack → Getting Started → Development → Project Structure → Contributing → License
