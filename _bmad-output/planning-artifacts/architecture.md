## Project Context Analysis

### Requirements Overview

**Functional Requirements (39 FRs, 7 categories):**

| Category | FRs | Architectural Weight |
|----------|-----|---------------------|
| Knowledge Capture & Persistence | FR1-FR7 | **Critical** — foundation of entire system; determines knowledge storage format, elicitation protocol design, and cross-session persistence mechanism |
| System Construction (Build Mode) | FR8-FR13 | **High** — must scaffold independent sub-projects, wire prompt chains, validate against acceptance criteria, and register into Execute Mode |
| Task Routing & Execution (Execute Mode) | FR14-FR18 | **High** — natural language task assignment, system identification, routing, end-to-end execution, internal error handling |
| Output Delivery & Validation | FR19-FR23 | **Medium** — staged delivery with approval gates, self-validation against acceptance criteria, override capability |
| System Diagnosis & Maintenance | FR24-FR29 | **Medium** — end-to-end pipeline inspection, root cause identification, fix proposal, guard rail addition |
| Knowledge Graph & Compounding | FR30-FR34 | **High** — cross-references, failure patterns, accumulated knowledge leveraging, dependency tracking |
| Platform Management | FR35-FR39 | **Medium** — system registry, unified CLI, health reporting, migration support, per-system dependency management |

**Non-Functional Requirements (13 NFRs, 4 categories):**

| Category | NFRs | Key Constraint |
|----------|------|----------------|
| Reliability & Stability | NFR1-NFR5 | Zero silent failures; state survives session restarts; sub-project isolation; no unvalidated output delivery |
| Integration | NFR6-NFR9 | Agent-readable markdown (no specialized parsers); independent dependency pattern; existing hook infrastructure; justfile orchestration |
| Security | NFR10-NFR11 | Credentials in env vars only; no sensitive client data in knowledge files |
| Performance | NFR12-NFR13 | CLI responsiveness (seconds, not minutes); knowledge graph queries performant at 10+ systems |

**Scale & Complexity:**

- Primary domain: CLI-based agent orchestration platform
- Complexity level: Medium (novel meta-system patterns, but single-user, no regulatory burden)
- Estimated architectural components: ~8-10 major components (knowledge store, build pipeline, execute router, validation engine, system registry, diagnosis engine, knowledge graph, CLI interface, hook integration layer)

### Technical Constraints & Dependencies

**Hard constraints from v0 brownfield state:**
- Bun runtime, TypeScript primary, Python hooks secondary
- ESM-only (`"type": "module"` everywhere)
- No shared workspace — each sub-project has independent dependencies
- justfile as the orchestration task runner
- Claude Code hooks/skills/agents as the agent runtime environment
- SQLite per-service for data persistence
- `library.yaml` auto-generated by `library_sync.py` — do not edit manually

**Hard constraints from PRD:**
- CLI-only orchestration — no GUI for Build/Execute/Evolution modes
- Knowledge stored as agent-optimized markdown (not BMAD format — lightweight, no workflow overhead)
- Engineer-as-CEO: human approval at all decision points
- Sub-project isolation by design — Build Mode must scaffold independent projects
- TypeScript/Node.js default for new systems built through Build Mode

**Integration constraints:**
- Must integrate with Claude Code's existing hook and session infrastructure (NFR8)
- Must follow existing justfile orchestration pattern (NFR9)
- Must follow existing submodule-style pattern for new systems (PRD)

### Cross-Cutting Concerns Identified

1. **Knowledge Persistence Across Sessions** — The core problem. Agent sessions are ephemeral; domain knowledge, quality criteria, and acceptance standards must survive session boundaries. Every mode depends on reliable knowledge access.

2. **System Registry & Discovery** — Execute Mode routing, Build Mode registration, Platform Management health checks, and Diagnosis all need a consistent way to know what systems exist, what they handle, and their current state.

3. **Self-Validation Pattern** — Build Mode captures acceptance criteria; Execute Mode enforces them; Diagnosis validates fixes against them. The validation mechanism must be consistent and domain-agnostic.

4. **Sub-Project Isolation** — Each system built through Adcelerate is independent (own deps, own config, own tests). The architecture must orchestrate across isolated boundaries without coupling.

5. **Claude Code Session as Runtime** — The orchestration layer doesn't run a persistent server. The agent session IS the runtime. Architecture must account for session-scoped execution with persistent state stored on disk.

## Starter Template Evaluation

### Primary Technology Domain

**CLI-based agent orchestration platform** — brownfield v0 → v1 upgrade. Not a greenfield build requiring a starter template.

### Starter Options Considered

| Option | Evaluation | Verdict |
|--------|-----------|---------|
| **CLI framework (oclif, commander)** | Would create a dedicated `adcelerate` binary. Adds a runtime layer between engineer and agent session. Duplicates what Claude Code already provides (natural language CLI, tool execution, session management). | **Rejected** — unnecessary abstraction |
| **Full-stack starter (T3, Next.js)** | Not applicable — v1 orchestration is CLI-only, not a web app. Existing web apps (observability, pinboard) already have their stacks. | **Rejected** — wrong domain |
| **No external starter — extend existing patterns** | v0 establishes all foundational technology choices. New v1 components integrate into existing monorepo structure using established conventions. | **Selected** |

### Selected Approach: Extend Existing Monorepo Patterns

**Rationale:** Adcelerate's runtime environment is the Claude Code agent session. The engineer interacts via natural language in the terminal; the agent reads knowledge files, executes tasks, and writes output. Adding a separate CLI binary or framework would create a redundant interface layer and split the orchestration logic between two systems. The justfile pattern already provides reproducible, named commands for anything that needs a stable entry point.

**Foundation — Existing Stack (No Changes):**

| Layer | Technology | Status |
|-------|-----------|--------|
| Runtime | Bun | Established |
| Language | TypeScript (strict, ESM) | Established |
| Task Runner | just (justfile) | Established |
| Agent Interface | Claude Code sessions (natural language CLI) | Established |
| Agent Tooling | Claude Code hooks (Python/uv), skills, agents | Established |
| Knowledge Format | Agent-optimized markdown + YAML | New (per NFR6) |
| Testing | Vitest | Established (autoCaption), extend to new components |
| Linting/Formatting | Biome | Established (autoCaption), extend to new components |
| Database (per-service) | SQLite via bun:sqlite | Established |

**New Conventions for v1 Components:**

- **Knowledge files:** Markdown with structured frontmatter — stored on disk, readable by agents without parsers (NFR6)
- **System registry:** YAML or markdown manifest — lists registered systems, their capabilities, and health state
- **Build Mode scaffolding:** justfile recipes + agent skills that create new sub-project directories following existing isolation pattern
- **Execute Mode routing:** Agent reads system registry, matches task to system, delegates execution
- **Validation:** Acceptance criteria stored as structured markdown per-system, enforced by agent logic during execution

**No initialization command required** — the foundation is the existing monorepo. New v1 directories and conventions are introduced incrementally as architecture decisions are made in subsequent steps.

## Core Architectural Decisions

### Decision Priority Analysis

**Critical Decisions (Block Implementation):**

- System Registry & Routing (registry format, task matching, lifecycle states)
- Build Mode Pipeline (implementation model, scaffolding, knowledge capture output)

**Important Decisions (Shape Architecture):**
- Execution & Validation (staged delivery, validation engine, error handling)
- Platform Structure & Integration (directory layout, hook evolution, migration strategy)

**Deferred Decisions (Post-MVP):**
- Evolution Mode automation (Phase 2)
- Performance Memory and recipe auto-generation (Phase 2)
- Discord command center integration (Phase 3)
- Parallel pipeline execution (Phase 3)

### Knowledge Architecture

**Decision 1a: Directory Structure — Hybrid (central index + co-located knowledge)**

Each system maintains its own knowledge directory co-located with its code. A central `knowledge/` directory at the monorepo root holds the knowledge graph and cross-system artifacts. The central layer handles discovery and routing; the system layer holds substance.

- Rationale: Fast routing at root level, deep knowledge lives with the code it describes, respects sub-project isolation pattern
- Affects: FR5 (persistence), FR6 (loading), FR30-FR34 (knowledge graph), all modes

**Decision 1b: File Schema — Multi-file with index per system**

Each system's knowledge directory contains standardized files (`index.md`, `domain.md`, `acceptance-criteria.md`, `dependencies.md`, `history.md`). The `index.md` summarizes the system and points to detailed files. Agents read the index first, then load specific files as needed.

- Rationale: Agent-efficient (selective loading), consistent with BMAD sharding pattern already in use
- Affects: FR5-FR6, NFR6 (agent-readable without parsers)

**Decision 1c: Acceptance Criteria Format — Tiered (hard gates + soft judgment)**

Each system's `acceptance-criteria.md` contains two sections: **Hard Gates** (structured checklist, binary pass/fail — things that MUST be true) and **Soft Criteria** (prose-based quality guidance — things that SHOULD be true, surfaced at approval gates for human judgment).

- Rationale: Hard gates enable closed-loop automation; soft criteria preserve engineer's expert judgment; maps to engineer-as-CEO principle and FR23 (override capability)
- Affects: FR3 (formalization), FR21 (self-validation), FR23 (override), NFR4 (no unvalidated output)

**Decision 1d: Knowledge Graph — Central graph file + embedded references**

A `knowledge/graph.yaml` at the root maps all system-to-system relationships, shared dependencies, failure patterns, and cross-references. Individual system knowledge files also contain inline markdown links to related systems for in-context navigation.

- Rationale: Single-file topology queries for routing and impact analysis; inline references for contextual awareness during build/diagnosis sessions
- Affects: FR30-FR34 (knowledge graph requirements), FR33 (dependency change tracking)

### System Registry & Routing

**Decision 2a: Registry Format — YAML manifest + convention validation**

A `systems.yaml` file at the monorepo root (alongside `library.yaml`) is the authoritative registry of all systems. Build Mode registration and a validation hook verify that referenced system directories and knowledge files actually exist on disk.

- Rationale: Fast agent lookup from single file, filesystem as truth prevents stale entries, mirrors existing `library.yaml` pattern, satisfies NFR1 (no silent failures)
- Affects: FR12 (registration), FR15 (identification), FR35-FR37 (platform management)

**Decision 2b: Task Matching — Structured capabilities + natural language fallback**

Each system entry in `systems.yaml` declares structured capability fields (`task_types`, `input_types`, `output_types`, `domain_tags`) and a natural language `description`. The execute router tries structured matching first. On ambiguous matches, falls back to natural language reasoning. Ambiguous results are surfaced to the engineer for confirmation.

- Rationale: Deterministic routing for common cases, graceful handling of edge cases, ambiguity surfaced to engineer (engineer-as-CEO principle)
- Affects: FR14-FR16 (task routing), NFR12 (CLI responsiveness)

**Decision 2c: Lifecycle States — Four-state (building / active / degraded / broken)**

Systems transition through: `building → active` (on successful build validation), `active → degraded` (on partial failure), `active/degraded → broken` (on critical failure), any state → `active` (on successful fix). The `building` state prevents Execute Mode routing to incomplete systems.

- Rationale: Matches FR37 requirement, prevents routing to half-built systems, clear transition rules
- Affects: FR12 (registration), FR37 (health reporting), FR35 (system status)

### Build Mode Pipeline

**Decision 3a: Implementation — Skill-orchestrated agent pipeline**

Build Mode is a Claude Code skill (`.claude/skills/adcelerate-build/`) that defines the overall workflow: knowledge-capture interview, formalization, scaffolding, validation, registration. At compute-heavy stages, the skill launches specialized sub-agents (scaffolding agent, validation agent) that operate in their own context windows.

- Rationale: Skill layer handles the conversational interview (needs full context); agents handle parallelizable tasks (scaffolding, testing) independently; workflow is readable markdown; extensible by adding new specialist agents
- Affects: FR1-FR2 (knowledge capture), FR8-FR13 (system construction)

**Decision 3b: Scaffolding — Template with adaptive layers**

A `templates/system/` directory contains a minimal base template (package.json, tsconfig.json, justfile, biome.json, knowledge directory structure with empty index.md). Build Mode copies this base, then the scaffolding agent adds system-specific files (prompt chains, model configuration, pipeline logic) based on captured knowledge.

- Rationale: Template enforces platform conventions (dependency management, knowledge format, testing setup); agent generates domain-specific logic; template can evolve and all future builds inherit updates
- Affects: FR9 (scaffolding), NFR7 (independent dependency pattern), NFR9 (justfile orchestration)

**Decision 3c: Knowledge Capture Output — Progressive write with formalization pass**

During the knowledge-capture interview, the skill writes incrementally to the system's knowledge directory as each topic is captured. After the interview completes, a formalization pass reviews all files, ensures consistency, structures acceptance criteria into hard gates and soft criteria, and updates the system's index.md. The engineer reviews formalized output before the build proceeds.

- Rationale: Crash-safe (incremental writes survive session failures), formalization ensures agent-optimized output quality, engineer review gate before build respects approval pattern
- Affects: FR1-FR7 (knowledge capture and persistence), FR13 (engineer approval)

### Execution & Validation

**Decision 4a: Staged Delivery — System-defined stages with minimal contract**

Each system defines its own stages in its knowledge index (count, names, what each stage produces). The contract requires: at least one stage, every stage produces reviewable output, final stage includes validation. The execute router reads stage definitions and manages approval flow generically.

- Rationale: Systems are too diverse for one-size-fits-all staging; minimal contract gives router enough structure without constraining domain-specific workflows
- Affects: FR19-FR20 (staged delivery with approval gates)

**Decision 4b: Validation Engine — Two-pass (automated hard gates + independent review agent)**

First pass: the executing agent checks output against hard gate criteria (binary pass/fail). Second pass: if hard gates pass, a separate validation agent reviews output against soft criteria with a fresh context window and produces a validation report. The engineer sees output + validation report at the approval gate.

- Rationale: Hard gates are cheap inline checks; soft criteria benefit from unbiased independent review; implements Two-Judge Standard (validation agent = first judge, engineer = second judge)
- Affects: FR21-FR23 (validation and override), NFR4 (no unvalidated output delivery)

**Decision 4c: Error Handling — Tiered with retry budget**

Three tiers: (1) **Inline recovery** — executing agent handles transient errors within a retry budget (e.g., 3 attempts). (2) **Diagnostic escalation** — if budget exhausted, agent produces structured error report (what failed, what was tried, suspected root cause) and escalates to engineer with actionable options. (3) **System-level diagnosis** — if engineer chooses, the Diagnosis pipeline (FR24-FR29) kicks in as a separate workflow.

- Rationale: Prevents silent infinite retries and unnecessary interruptions; retry budget bounds agent effort; structured escalation gives engineer actionable information; clean handoff to diagnosis workflow
- Affects: FR18 (internal error handling), FR24-FR29 (diagnosis pipeline), NFR1 (no silent failures)

### Platform Structure & Integration

**Decision 5a: Directory Layout — Purposeful root-level placement**

New root-level additions:
- `systems.yaml` — system registry (alongside `library.yaml`)
- `templates/` — scaffolding base templates
- `knowledge/` — central knowledge graph and cross-system artifacts

Individual system knowledge directories remain co-located within each system's sub-project.

- Rationale: Natural placement alongside existing root concerns, minimal root clutter (2-3 new entries), logical grouping with existing catalog files
- Affects: All modes (discovery, routing, building)

**Decision 5b: Hook Evolution — Composable (thin hooks + intelligent skills/agents)**

Existing hooks remain focused on observability and session lifecycle. They gain lightweight event emissions for v1-relevant moments (system invocation, validation completion). All orchestration intelligence lives in skills and agents, not hooks. The existing observability dashboard automatically captures v1 events without modification.

- Rationale: Hooks must stay fast (fire on every tool use); orchestration logic needs full context windows (skills/agents); existing observability pipeline inherits v1 visibility for free
- Affects: NFR8 (existing hook infrastructure), observability dashboard

**Decision 5c: Migration Strategy — In-place registration with knowledge bridge**

Pinboard and autoCaption remain as git submodules in their current locations. Migration means: (1) create a `knowledge/` directory inside each submodule, (2) register in `systems.yaml`, (3) add cross-reference entries in central `knowledge/graph.yaml`. No file moves, no history breaks.

- Rationale: Honors "migrate as-is" from PRD; preserves git history; pure validation of Build Mode on real unmodified systems; minimal structural disruption
- Affects: FR38 (migration without pre-modification), PRD validation target

## Implementation Patterns & Consistency Rules

### Pattern Categories Defined

**6 conflict areas identified** where AI agents working on v1 components could make inconsistent choices. Existing code patterns (47 rules in project-context.md) remain authoritative for established sub-projects — these patterns cover only new v1 conventions.

### Knowledge File Conventions

**Frontmatter schema (all knowledge files):**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `system` | string (kebab-case) | Yes | System identifier matching systems.yaml key |
| `type` | enum | Yes | `domain`, `acceptance-criteria`, `dependencies`, `history`, `index` |
| `version` | integer | Yes | Incremented on major restructuring |
| `lastUpdated` | date | Yes | ISO date |
| `lastUpdatedBy` | enum | Yes | `build-mode`, `engineer`, `diagnosis` |

**File naming:** Lowercase kebab-case — `domain.md`, `acceptance-criteria.md`, `dependencies.md`, `history.md`, `index.md`

**Directory:** Always `knowledge/` within the system directory

**Acceptance criteria structure:**
- Hard gates: Markdown checklist format (`- [ ] Criteria`) — parseable for pass/fail
- Soft criteria: Prose paragraphs with **bold emphasis** on key quality signals

### System Registry Schema

**systems.yaml entry format:**

| Field | Required | Type | Description |
|-------|----------|------|-------------|
| `name` | Yes | string | Display name |
| `path` | Yes | string | Relative path from monorepo root |
| `status` | Yes | enum | `building`, `active`, `degraded`, `broken` |
| `description` | Yes | string | Natural language description for NL fallback matching |
| `task_types` | Yes | string[] | Structured capability tags for primary matching |
| `knowledge_path` | Yes | string | Path to system's knowledge directory |
| `input_types` | No | string[] | What the system accepts |
| `output_types` | No | string[] | What the system produces |
| `domain_tags` | No | string[] | Domain categorization |
| `entry_point` | No | string | Main file path |
| `justfile` | No | boolean | Whether system has its own justfile |
| `stages` | No | object[] | Execute Mode stage definitions (`name`, `produces`) |
| `registered_at` | No | date | Registration timestamp |
| `last_validated` | No | date | Last successful validation |

**System identifier:** kebab-case key in systems.yaml (e.g., `autocaption`, `storyboard`)

### Knowledge Graph Format

**Central `knowledge/graph.yaml` structure:**

- **Per-system entries:** `depends_on` (with type: `runtime`/`build`/`optional`), `shared_models`, `related_systems`, `failure_patterns`
- **Cross-system relationships:** `from`, `to`, `type` (`pipeline`/`shared-dependency`/`shared-model`), `description`
- **Failure patterns:** `id` (fp-NNN), `description`, `diagnosed` date, `resolution`, `affects` array
- **Metadata:** `lastUpdated`, `systemCount`
- **Flat structure** — no deep nesting, optimized for agent scanning (NFR13)

### Event Naming Conventions

**v1 observability events:**

Convention: `adcelerate.<mode>.<action>` — dot-delimited, lowercase, snake_case for multi-word actions

| Mode | Events |
|------|--------|
| `build` | `started`, `knowledge_captured`, `scaffolding_started`, `validation_passed`, `registration_complete` |
| `execute` | `task_received`, `system_matched`, `stage_completed`, `validation_passed`, `delivered` |
| `diagnose` | `started`, `root_cause_found`, `fix_proposed`, `fix_applied` |
| `error` | `inline_retry`, `budget_exhausted`, `escalated` |

**Payload contract:** Every event includes `{ system_id, timestamp, session_id }` plus action-specific data.

**Existing events untouched** — v1 events are additive. Dashboard filters by `adcelerate.*` prefix.

### Structured Report Formats

**Error Report (diagnostic escalation):**
- Frontmatter: `type: error-report`, `system`, `stage`, `timestamp`, `retry_budget`, `retries_used`
- Body sections: Error Summary, What Was Tried (numbered attempts with results), Suspected Root Cause, Recommended Actions (checklist with retry/diagnose/abort options)

**Validation Report (review agent output):**
- Frontmatter: `type: validation-report`, `system`, `stage`, `timestamp`, `hard_gates_passed` (boolean), `soft_criteria_flags` (count)
- Body sections: Hard Gates (checklist with pass/fail), Soft Criteria Review (per-criterion assessment with warning markers for flags), Recommendation

**Format rules:**
- `[x]`/`[ ]` for hard gate checklists
- Warning marker for soft criteria flags
- Machine-parseable frontmatter + human-readable body

### Skill & Agent Naming

**Directory conventions:**
- Skills: `.claude/skills/adcelerate-<mode>/` (e.g., `adcelerate-build`, `adcelerate-execute`, `adcelerate-diagnose`)
- Agents: `.claude/agents/adcelerate-<role>` (e.g., `adcelerate-scaffolder`, `adcelerate-validator`, `adcelerate-formalizer`)

**Naming rules:**
- Prefix: always `adcelerate-`
- Skills: verb or mode name (orchestrate conversations with engineer)
- Agents: role noun (perform independent work in own context)
- No version numbers in names — versioning via git
- Discovery: glob `.claude/skills/adcelerate-*` or `.claude/agents/adcelerate-*`

**Namespace isolation:** BMAD skills keep `bmad-` prefix, marketing skills keep their names. No collision.

### Enforcement Guidelines

**All AI agents working on Adcelerate v1 MUST:**

1. Read `project-context.md` for established code patterns (47 rules) before implementing
2. Follow knowledge file frontmatter schema exactly — missing fields cause agent confusion downstream
3. Use `systems.yaml` as the single source of truth for system discovery — never hardcode system paths
4. Use the `adcelerate.*` event naming convention for all v1 observability events
5. Produce error reports and validation reports in the specified format — the engineer and other agents depend on consistent structure
6. Name new skills/agents with the `adcelerate-` prefix — discovery depends on it

**Anti-patterns:**
- Creating knowledge files without frontmatter
- Adding systems by modifying code instead of registering in `systems.yaml`
- Inventing ad-hoc error or validation report formats
- Placing knowledge files outside the `knowledge/` directory convention
- Using camelCase or PascalCase for knowledge file names

### Architectural Boundaries

**Orchestration Boundary:**
- Skills (`.claude/skills/adcelerate-*`) own the workflow logic — they decide what happens and in what order
- Agents (`.claude/agents/adcelerate-*`) own the execution logic — they do the work within their scope
- Skills invoke agents, never the reverse
- Communication: skill launches agent with prompt → agent returns result

**Knowledge Boundary:**
- Central `knowledge/` owns cross-system topology (graph.yaml, shared patterns)
- System-level `[system]/knowledge/` owns system-specific domain knowledge and criteria
- Central reads but never writes to system-level knowledge (Build Mode writes to system level, then updates central graph)
- System-level knowledge files reference central graph for cross-system context via inline links

**Registry Boundary:**
- `systems.yaml` is the single source of truth for system discovery
- Only Build Mode registration and Diagnosis workflows write to it
- Execute Mode, validation, and platform management read from it
- Filesystem validation ensures registry entries match actual directories

**Sub-Project Isolation Boundary:**
- Each system (autoCaption, pinboard, future systems) has its own dependencies, tests, justfile
- No imports across sub-project boundaries
- Communication between systems happens through the orchestration layer (skills/agents), never through direct code imports
- Shared conventions (knowledge format, registry schema) are documented, not enforced by shared code

### Data Flow

```
Engineer (CLI prompt)
  → Claude Code Session
    → adcelerate-execute skill reads systems.yaml
      → Matches task to system via structured capabilities
        → Routes to target system
          → Executing agent reads [system]/knowledge/ for context
            → Produces staged output
              → Hard gate check (inline, reads acceptance-criteria.md)
                → adcelerate-validator agent (soft criteria review)
                  → Validation report returned
                    → Engineer sees output + report at approval gate
                      → Approved → delivered
                      → Rejected → back to executing agent with feedback
```

```
Engineer (Build Mode)
  → adcelerate-build skill starts interview
    → Progressive writes to [system]/knowledge/
      → adcelerate-formalizer agent reviews & structures
        → Engineer reviews formalized knowledge
          → adcelerate-scaffolder agent creates sub-project from template
            → Validation tests run
              → System registered in systems.yaml
                → knowledge/graph.yaml updated with new system
```
