# The Drive approval gate is a durable control-plane checkpoint, not a Step/Run state

In **Drive** mode the Canvas launches a Flow, and some actions (spend, publish, overwriting `client/`) need operator consent before they run. We had to decide *where* that consent happens, *what state* the work is in while it waits, *what trips it*, and *how long it lives*. The shape we picked treats approval as a **control-plane checkpoint** layered over execution — never as a new state inside the data-plane Step/Run lifecycle.

**Decision:**

1. **One action surface — the left-zone chat.** Approve/reject happens only via the Orchestration Agent's `canUseTool` gate in the Console's left zone. The affected Canvas nodes show a **non-interactive** "⏸ awaiting approval" overlay that deep-links to the pending chat prompt — *not* a second approve button. One action surface = one control-plane writer for the decision, which also pre-empts the Reflect/Drive identity collision (Item 9).

2. **Not a lifecycle state.** Approval-pending is a **derived overlay**, not a Step or Run state. The Step lifecycle (`queued → running → succeeded/failed`) stays honest about *execution*; `queued` keeps meaning "committed, awaiting a slot." The gated Step is **born** (emitted → `queued`) only on approve; on reject it is never dispatched (cascade: the queued tail is never dispatched) — nothing to roll back. Two render forms: overlay on **existing** nodes for a cascade, **ghost/banner node** for a fresh dispatch. This mirrors the existing *Stale* "blocked: upstream failed" derived view.

3. **What trips it keys on effect class**, not tool identity: pure reads auto-allow; spend-incurring dispatch gates only when it's a cascade **above 2 nodes** *or* crosses a **budget line** (folding the existing *Stale* ">2 node" confirm gate and the budget-headroom warning into one rule); external/irreversible effects (publish, overwrite `client/` Reference data, delete) **always** gate. v1 ships fixed defaults with thresholds as named constants — no operator settings UI yet.

4. **Durable, no timeout.** The trip (`approval-requested`) and the resolution (`approval-granted`/`approval-denied`) are each **control-plane events** in the Substrate's ordered log — so the gate survives reload / Board-reopen and carries provenance (which operator, when). There is no auto-resolution timer (auto-approve would spend without consent; auto-reject would silently kill intended work); it waits indefinitely. The Run stays plain `running` (idle-by-reason-of-gate) — no `awaiting-approval` Run status. If the hosting Orchestration Agent session dies while a gate is pending, reconnect **re-arms** the `canUseTool` wait from the durable event; if it cannot resume, the Run terminates `cancelled`.

**Why this shape:** approval is fundamentally *control plane* — a decision about whether execution may start — while Steps/Runs are *data plane*, honest about execution that is actually happening. Bolting an `awaiting-approval` state onto either lifecycle would force every data-plane consumer (Emitter contract, Canvas renderer, reconciliation) to special-case a state that isn't execution. Keeping the gate as a durable, event-sourced control-plane overlay reuses the machinery the Substrate already has (ordered log, control-plane events, derived "blocked" views) and gives spend/irreversible decisions a real audit trail.

**Considered and rejected:**
- **Inline-on-node as the action surface** — more "direct manipulation," but it makes a Canvas node a control-plane *writer*, splits approval state across two surfaces, has no single home for set-approvals (a 5-node cascade), and reintroduces the dual-writer race of Item 9.
- **First-class `awaiting-approval` Step/Run state** — tidy to render as one state machine, but pollutes the data plane with a control-plane concept and muddies what `queued`/`running` mean.
- **Blanket-approve every dispatch** — safe but collapses Drive into Reflect-with-extra-clicks; **full autopilot (gate nothing)** removes the human from spend/irreversible decisions, defeating the gate's purpose.
- **Ephemeral session-bound gate** — simpler (no event), but closing the laptop silently abandons a pending decision and leaves no audit of who approved what; unacceptable for anything touching spend or `client/`.
- **Auto-resolution timeout** — either auto-approve (spends without consent) or auto-reject (kills intended work); both are wrong, so the gate waits indefinitely.

**Consequence:** the Substrate's control-plane event vocabulary grows three event types (`approval-requested`, `approval-granted`, `approval-denied`), folded by the projection like other control-plane events. The Canvas gains a derived "awaiting approval" overlay state (no new emitted Step state). Drive's `canUseTool` policy is the enforcement point and must be mirrored to the log on both trip and resolution. The session-death re-arm path is the first concrete dependency on Item 9 (Reflect/Drive identity), addressed there.
