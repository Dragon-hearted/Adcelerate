# Adcelerate

A monorepo of independent AI marketing/media systems, orchestrated through Claude Code skills and agents. This glossary covers the cross-cutting vocabulary for **execution visibility** — the canvas and command-center observability work.

## Language

**Flow**:
One end-to-end creative process exposed by a system (e.g. a post-board carousel build, a scene-board storyboard, an image-engine batch). Driven by a skill in the backend.
_Avoid_: pipeline, workflow, skill-run

**Run**:
A single invocation of a Flow. A Run emits Steps as it executes, and is itself bracketed by an explicit **Run lifecycle**: the Emitter fires `run.started` when the Flow's skill is invoked and `run.completed(status)` when it terminates — completion is *emitted*, never inferred from Step states or `sessions.endedAt` (which is absent for Reflect-mode external CLI Runs). A cascade (see *Stale*) is its own Run on the same Board. See *Run lifecycle*.
_Avoid_: session, job, execution

**Run lifecycle**:
The explicit bracket around a Run: `run.started` … `run.completed(status)`, emitted by the Flow's skill wrapper at invocation entry/exit (parallel to the per-Step *Step lifecycle*). Terminal `status` is **non-binary**: `completed` (all Steps succeeded) / `completed-with-failures` (Run finished, ≥1 Step `failed`) / `failed` (the Run itself crashed) / `cancelled` (stopped before terminating — the hook for mid-cascade cancellation). **Run-done ≠ Board-done**: `run.completed` may trigger an auto-save/notify on the *Board* (see *Board*), but that is a convenience, not a "finished" signal — a multi-Run Board is not done when its first Run completes.
_Avoid_: status, done, finished (the discrete state is the terminal `status` value), session-end (`sessions.endedAt` is a Drive-only artifact, not the lifecycle)

**Step**:
A single inspectable unit of work emitted by a Run. Steps are **hierarchical**: a coarse Stage (from `systems.yaml`) contains finer Ops (one model call / inpaint / upscale), and the Canvas lets you zoom and edit at either level. Steps connect into a Step Graph. A Step is a *logical slot*, not an execution: it has a **stable identity** (a semantic step-key) that persists across re-runs, so all of a Step's Branches attach to the same Step rather than appearing as unrelated Steps. That identity is a **slot id carried forward** (image-engine's `sceneId`), assigned once and reused — *never* derived from the item's content, because editing a prompt must create a Branch on the *same* Step, not a new one. A re-plan is a slot-id **diff** (keep/edit/add/remove), not a blind regeneration; a slot dropped from the current plan becomes *Orphaned*, never deleted. See *Orphaned*.
A Step also has an observable **lifecycle**: `queued` → `running` (with an optional `retrying(n/max)` sub-state) → `succeeded` | `failed`. The Emitter reports these transitions as they happen — *not* a single terminal event — so the Canvas renders progress *within* a Step, not just its final artifact. Because image-engine's providers are opaque (no percentage, no mid-render callback), the honest signals are coarse lifecycle states plus a client-side elapsed timer, never a fabricated progress bar. See *Step lifecycle*.
_Avoid_: stage (a Stage is one *level* of Step — the coarse phase), node (a node is the *visual* rendering of a Step on the Canvas)

**Step lifecycle**:
The state machine a single Step passes through: `queued` (created, awaiting a concurrency slot) → `running` (dispatched to a provider; duration opaque) → `succeeded` (artifact landed) or `failed` (provider error/timeout/non-zero exit). `running` may carry a `retrying(n/max)` sub-state (e.g. a WisGate `attempt 2/3` after a 429). The Canvas renders each state honestly: skeleton card while `queued`, a pulsing "generating…" with a **client-side** elapsed timer while `running`, a retry chip on `retrying`, the artifact on `succeeded`, an error card with retry affordance on `failed`. `failed` is the single source of failure semantics — there is no separate failure model.
_Avoid_: status, progress (this is a discrete state machine, not a percentage)

**Step Graph**:
The DAG/tree of Steps a Run produces, rendered visually on the Canvas.
_Avoid_: trace-tree, node graph

**Console**:
The single operator surface for running creative Flows. Two zones: a **left zone** (chat + history + lightweight agent-activity summary + a prompt box with model/param controls) and a **right zone** (the Canvas). One app, two zones, one Substrate — this replaces the earlier "two separate surfaces" model. System-*building* stays in the CLI (Claude Code); the Console exists for *running generation Flows*.
_Avoid_: dashboard, IDE, app

**Canvas**:
The Console's **right zone**: the visual surface that renders a Run's Step Graph and lets you inspect, edit, and re-run from a chosen Step. Generations land here one by one as a Run executes. The Canvas holds two distinct kinds of objects: **Steps** (Run-emitted, immutable, branchable) and **user-authored material** (uploads, text notes, and Reference Groups — free-form, not from any Run). The Canvas is the live *surface*; what persists across sessions is the **Board** (see below) — the Canvas is the view, the Board is the saved thing.
_Avoid_: board (a Board is the persisted workspace; the Canvas is its live rendering), editor

**Board**:
The unit of persistence — a named, durable workspace the operator saves and reopens. A Board **references one-or-more Runs** (reusing the existing session/event-log replay for their Step Graphs), and **owns** the two things that have no event-sourced home: the **canvas layout** (node positions, zoom, pins) and the **user-authored material** (uploads, text notes, Reference Groups). One Board can span multiple Runs (e.g. a scene-board storyboard Run plus an image-engine batch Run launched off one of its frames). v1 default is one-Run-per-board, but the schema is many-Runs-per-board from the start. Runs stay immutable event-sourced content; the Board is the savable/nameable/reopenable container layered on top.
_Avoid_: project, workspace, session (a session is one Run's backing store), canvas (the Canvas is the Board's live rendering)

**Reference Group**:
A first-class Substrate object: a user-authored bundle of image refs (and optional text intent) created by selecting/grouping/collaging items on the Canvas. Fed to a Run as **structured input** — separate refs by default (preserves per-image fidelity), with a **flatten toggle** that collapses the group into a single mood-board image when a flow wants whole-vibe style transfer. When an un-flattened group exceeds a provider's reference cap, it **partial-flattens**: **pinned** refs (the ones you mark as hero) are kept sharp and separate, filling slots first, and the overflow tail is collapsed into a single image occupying the last slot — no ref is discarded, and the Canvas labels the split honestly. That collapse is a **deterministic local montage** (`sharp`-tiled), **never a provider call** — so it can't 402, time out, or drift, and it keeps every overflow ref visible (an AI blend would destroy per-ref detail). Pinned refs are sacred: if the pinned set alone exceeds the cap, the Run blocks rather than silently dropping one. The **cap is not a constant** — it's a descriptor resolved at dispatch, `capFor(provider, model, modality) → { total, byCategory? }` (default Higgsfield route ≈ `{total: 8}`; WisGate ≈ `{total: 14, byCategory: {objects: 6, humans: 5}}`); partial-flatten triggers only when the *resolved* cap is exceeded, so most groups on the default model pass through untouched. Typed sub-caps are carried but not enforced in v1. Reusable across Runs and tracked in Lineage. The collage is the *view*; the structured set (or partial-flattened set) is the *payload*.
_Avoid_: collage, mood board (those are views of a Reference Group), selection

**Reflect**:
Canvas mode where flows run via their own skills and the Canvas only *reads* the emitted Steps (with the ability to fork/edit/re-run from one). The starting mode.
_Avoid_: read-only, mirror

**Drive**:
Later Canvas mode where the Canvas itself *launches* a Flow — by invoking the backend skill, never by reimplementing it.
_Avoid_: control, orchestrate

**Approval Gate**:
The control-plane checkpoint where a **Drive** Run pauses for operator consent before a spend-incurring or irreversible action. Surfaced and actioned **only in the Console's left-zone chat** (the Orchestration Agent's `canUseTool` mechanism); the affected Canvas nodes show a linked, **non-interactive** "⏸ awaiting approval" overlay that deep-links to the pending chat prompt — **never a second approve button** (one action surface = one control-plane writer, which also pre-empts the Reflect/Drive identity collision). The Gate is **not a Step or Run lifecycle state**: it's a *derived overlay* (same discipline as a *Stale* "blocked: upstream failed" view), and the gated Step is only **born** (emitted → `queued`) on approve; on reject it is never dispatched — for a cascade, the queued tail is never dispatched — so there is nothing to roll back. Two render forms: an overlay on **existing** nodes for a cascade ("these will be regenerated"), a **ghost/banner node** for a fresh dispatch (node doesn't exist yet). What trips it keys on **effect class**: pure reads auto-allow; spend-incurring dispatch gates only when it's a cascade **above 2 nodes** *or* crosses a **budget line** (folding the *Stale* ">2 node" confirm + budget-headroom warning into one rule); external/irreversible effects (publish, overwrite `client/` *Reference data*, delete) **always** gate. v1 ships **fixed defaults** (thresholds as named constants), not an operator settings UI. The Gate is **durable**: the trip (`approval-requested`) and the resolution (`approval-granted`/`approval-denied`) are each **control-plane events** in the ordered log — so it survives reload / Board-reopen and carries **provenance** (which operator, when). There is **no timeout** (auto-approve would spend without consent; auto-reject would kill intended work): it waits indefinitely, with the Run staying plain `running` (idle-by-reason-of-gate, not a new status). If the hosting Orchestration Agent session dies while pending, reconnect **re-arms** the `canUseTool` wait from the durable event; if it cannot resume, the Run ends `cancelled` and the overlay reads "Run ended before approval — re-drive to retry."
_Avoid_: confirm dialog, modal, permission prompt, canUseTool (that's the SDK mechanism; the Gate is the durable cross-zone concept)

**Branch**:
An immutable fork of a Step's output. Created either by editing a Step's prompt/params (the agent regenerates) or by supplying a replacement artifact yourself. Every Branch carries a **provenance** flag — *agent-generated* or *human-authored* — and exactly one Branch per Step is the **active** one the Flow resumes from. The original Step and all sibling Branches are preserved; nothing is mutated in place. Re-running cascades from the active Branch downstream. The active pointer is **event-sourced, not a mutable cell**: activation is an `activate` event folded by the projection in ingest order, and resolution is **human-sticky** — an agent cascade auto-activation never overrides a human active-selection made at-or-after the cascade's trigger (the regenerated agent Branch lands in Lineage as non-active, with a "newer regeneration available" affordance). Operator active-flips reach the Substrate as **control-plane** events (via the orchestrator), distinct from system Emitter (data-plane) events but sharing one ordered log.
_Avoid_: variant, version, fork (fork is the verb), override

**Lineage**:
The tree of Branches accumulated across a Run's editing history, rendered on the Canvas so variations can be compared side-by-side, each Branch's provenance (agent vs human) visible.
_Avoid_: history, version tree

**Stale**:
A flag on a Branch whose upstream dependency changed and was re-run. The cascade is **provenance-aware**: it auto-regenerates *agent-generated* downstream Branches, but leaves *human-authored* ones untouched and marks them Stale — preserved, flagged, awaiting your decision (regenerate vs keep-and-reconcile). Human work is never overwritten silently. A cascade is its own Run: it runs **async** (rendered live on the Canvas), shows a **confirm gate above 2 nodes** (count + per-provider/model breakdown + a budget-headroom warning — never a fabricated dollar total, since cost is uncomputable pre-run), **cancels cooperatively** (stop dispatching the queued tail; let in-flight gens finish since their spend is already committed; Run ends `cancelled`), and on a `failed` node leaves its downstream subtree undispatched — rendered as a **derived** "blocked: upstream failed" view, not a new emitted state.
_Avoid_: dirty, outdated, invalid

**Orphaned**:
A flag on a Step whose slot was **removed from the current plan** when the agent re-planned (e.g. a batch item the agent no longer emits). Like *Stale*, the Step and its Branches/Lineage are **preserved, not deleted** (immutability, see *Branch*) — it greys out on the Canvas labelled "removed from current plan," awaiting your decision (restore vs discard). Distinct from *Stale* (whose cause is an upstream change, not a plan change); both share the "preserve + flag, never silently drop" rule. Requires the slot-id (`sceneId`) to be stable across re-plans — re-plans are a keep/edit/add/remove **diff**, so a continued slot keeps its identity and only genuinely-dropped slots orphan.
_Avoid_: deleted, removed, dropped (the Step is kept, only flagged), stale (that's the upstream-change flag)

**Slot Identity**:
How a Step's logical identity stays stable and collision-free across the **Reflect** (CLI-launched) and **Drive** (Console-launched) modes that share one ingest. At **emit time** a Step is identified by **`(runId, step-key)`** — the only thing unambiguous at the source, since a producer (especially the CLI) knows its own `runId` but *not* Boards (a Console concept applied after the fact). So two independent Runs of the same Flow that both mint step-key `s1` are distinct Substrate Steps `(A,s1)` / `(B,s1)` and **can never silently fuse** in the monotonic fold. The *desired* stacking — re-running a slot stacks a new **Branch** onto one logical Step — happens one layer up in the **Board projection**, which maps multiple Runs' Steps onto one logical slot **only** when they are co-members of the same Board *and* share a **`(producerSystem, slot-id)`** key (never a bare step-key — so a scene-board `s1` and an image-engine `s1` cannot cross-fuse on a heterogeneous Board; cross-system relationships are *edges*, not merges). That merge is **automatic but visible** (Lineage tags each stacked Branch with its Run + provenance), gated by Board co-membership — which is itself an explicit operator act: a Reflect Run is **born Board-less**, appears in Console history on `run.started`, and joins a Board only via "open into Board" (the producer never learns Boards exist). Concurrency is bounded by a soft **Execution Lease**: a slot is "busy" while it holds a `queued`/`running`/`retrying` Branch (a *derived* read over the *Step lifecycle*, not new state), and a Drive that targets a leased slot **blocks at the Approval Gate** ("slot busy in Run X") rather than double-dispatching; the lease releases on terminal. The resolved active Branch still follows the *Branch* human-sticky rule.
_Avoid_: step id (the Substrate id is the pair `(runId, step-key)`; the step-key alone is not the identity), GUID, primary key

**Spawn Tree**:
The runtime tree of agents and sub-agents an agent Run produces, with per-agent tokens, tool calls, and runtime. The agent-flow analogue of a Step Graph. Lives as an **expandable panel inside the Console's left zone** (not a separate surface); the chat shows only a lightweight activity summary, and you open the panel for the dense telemetry.
_Avoid_: agent tree, call tree

**Orchestration Agent**:
The persistent Claude Agent SDK session behind the Console's left zone. It holds the conversation and approval state, and dispatches Flows by *invoking skills* (`adcelerate-execute`, then the matched system's own skill) — never by reimplementing flow logic. Enabled by passing the embedded SDK its dedicated `skills` option (keeping `settingSources: []`, so the `canUseTool` approval gate stays intact). Distinct from the **orchestrator**, which is the Bun server *process* (`@command-center/orchestrator`) that spawns and hosts these agent sessions.
_Avoid_: orchestrator (that's the server), bot, assistant, copilot

**Substrate**:
The shared spine the Console consumes — the Run/Step/Graph contract, the live transport, the event store, and the **artifact store** — living in the command-center workspace (`@command-center/shared` + orchestrator ingest). The Canvas (right zone) and the Spawn Tree (left-zone panel) are two *zones* of one Console over one Substrate. The guiding principle: the **Substrate owns the durable record; systems are transient producers** — events are event-sourced, and Artifacts are snapshotted into the Substrate's own store so a saved Board never depends on a producing system staying alive. The Substrate admits **two producer classes** writing one ordered event log: system Emitters (the **data plane** — Steps, artifacts, Run lifecycle) and the Console itself (the **control plane** — operator actions that change durable state: active-flips, pins, notes, layout saves). Both are subject to the same dedupe/ordering on ingest.
_Avoid_: backend, core, platform

**Artifact**:
The byte payload a Step produces — for image-engine, the generated image file. The producing system stores its own copy (image-engine: `./uploads/{genId}`, served at `:3002/api/gallery/{id}/image`), but that copy is **incidental and uncontracted** (local-disk only, bound to the system being up, no retention guarantee). So on ingest the Substrate **snapshots the bytes once** into its own artifact store, keyed by Step/Branch id, and every Board references *that* Substrate-owned URL — never the producing system's. The system's copy is the source-at-generation-time; the Substrate's copy is canonical for anything saved.
_Avoid_: image, file, asset, blob (those are the raw bytes; the Artifact is the Substrate's durable, addressable record of them)

**Reference data**:
The cold, human-authored, curated half of storage — brand briefs, prompt libraries, client dossiers, positioning/voice/research — as opposed to the hot, machine-emitted **operational data** (Runs, Steps, Branches, Boards, Artifacts, token ledger) that lives in the **Substrate**. Reference data lives as a plain on-disk markdown/JSON file tree (`client/{slug}/` — backed up out-of-band, never committed; root `knowledge/` — git-versioned), which is *already* an Obsidian-openable vault. Obsidian is an **optional viewer** over those files — never a runtime dependency, never the system-of-record. `brand.json` stays canonical machine-readable JSON (consumed by `loadBrand()`); its `brand.md` twin is the human-readable companion. The boundary is by **nature** (curated vs generated), not by folder.
_Avoid_: knowledge base, vault (a vault is one *view* of Reference data), CMS, database (operational data is in the Substrate; Reference data is files)

**Emitter**:
A thin per-system helper that POSTs semantic Step events to the Substrate's ingest endpoint. Each event carries a **stable step-key** — the logical Step identity, reused verbatim on re-run — not just the transient `tool_use_id` of one execution; this is what lets Branches stack onto a single Step. The Emitter reports the full **Step lifecycle** — firing on dispatch (`running`), on any `retrying`, and on the terminal `succeeded`/`failed` — *not* a single event at completion; this is what makes the Canvas live rather than batch-at-end. A system emits Steps at its **own natural unit of work** (image-engine: one per generation; scene-board: one per Stage) and supplies edges from **its own declared dependencies** — grain is per-system, not uniform. Delivery is **at-least-once** (the Emitter retries failed POSTs) and stamps each event with a producer per-Run monotonic seq; the Substrate makes this safe by **idempotent, order-tolerant ingest** — dedupe on `(branch-id, lifecycle-state, retry-attempt)` via `INSERT OR IGNORE`, a **monotonic per-branch fold** (state never regresses, so reordering is harmless), edges from declared deps (not arrival), and **run-completion reconciliation** for dropped terminal events (a still-non-terminal Step at `run.completed` renders a derived "outcome unknown"). Correctness rests on dedupe + fold + reconcile, *not* on seq. Every event also carries an **`envelopeVersion`** (SemVer): because the Emitter is **copied** into each isolated submodule rather than shared (submodules can't import the workspace), copies drift — and once systems ship as independently-versioned, SHA-pinned submodules a cloner lazy-inits, that drift is a *public* contract. Ingest validates the envelope against a versioned zod schema, **upcasts** known-but-old versions within a **compat window** (`≥ minVersion`), and **rejects unknown versions loudly (4xx)** — deliberately distinct from the silent dedupe above (a duplicate is harmless; an unknown version is a contract break that must surface, not poison the canonical store). The canonical Emitter lives in `@command-center/packages/shared`; copies are kept in sync by a local sha256 recipe, never silently. Version skew is surfaced as the Console's two-tier **freshness signal** (soft "update available" / hard "too old — update required").
_Avoid_: reporter, publisher, hook
